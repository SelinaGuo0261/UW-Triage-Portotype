# 产品模块分工（4 人）

本文档按**产品模块**划分职责（不按前后端职位）。适用于当前 UW Triage 原型向正式产品演进时的协作边界说明。

---

## 1. Researcher 端

**范围**

- 只围绕**已发布快照 / 知识库（Knowledge Base）**的体验：浏览、检索、按向导走流程、结果页等。
- 消费的是「已发布且对研究员可见」的内容，不操作 Builder 侧草稿图。

**不负责**

- Builder 草稿编辑、未发布 flow 的编辑界面、发布前的校验修复（属模块 2）。

---

## 2. Builder — 手动编辑

**范围**

- **画布**：节点与连线、缩放/平移、手动调整布局、与图结构相关的交互与展示。
- **流程资产**：流程列表、打开/关闭、回收站（或等价生命周期）、元信息（标题、描述等）。
- **发布**：发布到 `PUBLIC` / `INTERNAL`（或产品内等价范围）；发布失败时的提示与引导。
- **与校验联动**：展示阻塞/警告类校验结果，并支持用户通过编辑把流程修到**可发布**状态。

**不负责**

- 从文件自动生成整张初稿图的算法与提示词优化（属模块 3）。

---

## 3. 自动建流程（file2flow）

**范围**

- 从**文件 / 粘贴文本**到**可编辑初稿**（进入 Builder 前或进入后的首版图）：解析、结构化、与内部图模型对齐。
- 若产品包含：**上传流程、进度反馈、失败重试、错误说明**等体验，默认归本模块；若与 Builder 壳强绑定（例如仅在「新建」弹窗内），可与模块 2 **共担**，需在迭代前约定接口与 UI 边界。

**不负责**

- 研究员端只读体验（属模块 1）；长期账号与权限（属模块 4）。

---

## 4. 用户系统

**范围**

- **账号与登录**（实现方式见下文「现阶段简化方案」）。
- **数据库与持久化**中与「用户、租户、归属」相关的模型与策略。
- **对象归属**：例如 flow 属于谁/哪个工作区。
- **两类分享（需在需求上区分清楚）**
  - **协作**：谁能编辑同一 flow（编辑权限、邀请协作者）。
  - **分发**：谁能看到**已发布**内容（与 Researcher / 知识库可见范围相关，需与模块 1、2 对齐）。
- **邀请链接与权限模型**：创建、撤销、过期、角色（Owner / Editor / Viewer 等，以产品定义为准）。

---

## 未来 SSO 与「现阶段做不了 SSO」时的简化 / 替代方案

长期目标若是 **SSO**（由学校/机构 IdP 统一登录：OIDC / SAML 等），当前阶段往往拿不到 IdP 客户端、回调域名审批或合规排期。此时**仍然值得**先做用户与权限相关能力，但应把「登录」做成**可替换的适配层**，把「应用内身份与授权」做成**稳定内核**，避免将来接 SSO 时推倒重来。

### 原则

1. **认证（Authentication）可插拔**：谁能证明「我是我」——今天用简化登录，明天换成 SSO，只换这一层。
2. **授权与归属（Authorization + tenancy）保持稳定**：`user`、`workspace/org`、`flow` 归属、`membership`、`flow_share` 等表结构与业务规则尽量不因登录方式而变。
3. **外部身份预留映射字段**：例如 `external_subject`（OIDC `sub`）或 `idp_issuer + sub` 联合唯一；现阶段可为空，接 SSO 后写入。

### 推荐的「简化、替代系统」长什么样

| 能力 | 现阶段（无 SSO） | 说明 |
|------|------------------|------|
| 注册 / 登录 | **邮箱 + 密码** 或 **Magic link（邮件链接）** 或 **演示用「邀请码注册」** | 避免依赖学校 IdP；Magic link 可减少密码管理成本。 |
| 会话 | **HttpOnly Cookie + 服务端 Session** 或 **短期 JWT（httpOnly/same-site）** | 与后续 OIDC 回调后发 session 的形态一致，迁移平滑。 |
| 用户主键 | 内部 **`user_id`（UUID）** 不变 | SSO 登录后在服务端 `upsert`：按 `external_subject` 找到或创建用户，再建 session。 |
| 租户 | **单租户「个人空间」** 或 **简易 Workspace（名称 + 成员列表）** | 不必一上来做多组织 SAML 属性映射；Workspace 可先当「课题组」用。 |
| 协作分享 | **链接邀请 + 角色**（Editor / Viewer）+ 可选过期 | 与 IdP 无关；SSO 后仍保留。 |
| 分发（已发布可见） | 与模块 1、2 对齐：**PUBLIC / INTERNAL + 成员可见** | 逻辑不变，仅「谁算成员」随登录用户变化。 |
| 管理员 / 演示 | **环境变量配置的超级管理员邮箱** 或 **固定演示账号** | 仅用于开发/课堂演示，生产需收紧。 |

### 刻意不做或推迟的事（减少将来 SSO 返工）

- 不要把「权限判断」写死在校验「某个学校邮箱后缀」上（可作为**临时**规则，但要集中在策略模块，便于替换）。
- 不要自建复杂「社交账号绑定多 IdP」除非产品必需。
- 若机构将来强制 MFA，优先交给 **SSO IdP** 承担；本地密码阶段可只做基础强度与速率限制。

### 接 SSO 时的典型迁移路径（非重写）

1. 在登录页增加「通过学校账号登录」→ OIDC 授权码流程。
2. 回调中拿到 `sub` / `email`（以 IdP 为准），**匹配或创建**本地 `user`，写入 `external_subject`。
3. 对已有用户：可用 **邮箱匹配** 做一次性绑定（需防账号合并冲突，产品上要明确规则）。
4. 逐步关闭本地密码注册，或仅保留给外部合作方。

**结论**：现阶段做一个「**内部用户模型 + 会话 + 邀请协作 + 与 flow 绑定**」的系统，并用「邮箱密码或 Magic link」做登录，**仍然有意义**；与未来 SSO 的关系是**替换认证入口与外部身份映射**，而不是从零重做整个用户与权限域。

## 模块间依赖（简表）

| 依赖方向 | 说明 |
|----------|------|
| 1 ← 2 | 发布成功后才出现在研究员可见的知识库/快照体验中。 |
| 2 ← 3 | file2flow 产出初稿后，由 Builder 手动编辑与发布闭环接手。 |
| 2、3、4 | 校验规则、发布条件、权限与「谁能发布」需在接口与产品文档上对齐。 |
