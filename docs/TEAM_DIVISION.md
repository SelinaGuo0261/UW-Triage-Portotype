# 产品模块分工（5 条轨道）

本文档按**产品模块**划分职责（不按前后端职位）。适用于当前 UW Triage 原型向正式产品演进时的协作边界说明。

**当前人力分配**：模块 **1–4** 并行推进；模块 **5 用户系统暂缓**，不在本期排期。

---

## 1. Researcher 端 + UX Test

**范围**

- **Researcher 产品**：只围绕**已发布快照 / 知识库（Knowledge Base）**的体验——浏览、检索、按向导走流程、结果页等；消费「已发布且对研究员可见」的内容，不操作 Builder 草稿。
- **UX Test**：针对 Researcher 与跨模块关键路径的**可用性验证**——任务脚本（如「找到某 flow → 走完决策 → 到达某结果」）、问题记录、与模块 2 的预览/发布链路对齐的反馈（不写死实现细节，以可复现步骤与录屏/截图为佳）。

**不负责**

- Builder 画布编辑、流程列表/回收站实现、发布按钮逻辑（属模块 2）。
- file2flow 生成管线（属模块 3、4）。

---

## 2. Builder — 手动编辑（含 Preview）

**范围**

- **画布**：节点与连线、缩放/平移、手动调整布局、与图结构相关的交互与展示。
- **流程资产**：流程列表、打开/关闭、回收站（或等价生命周期）、元信息（标题、描述等）。
- **发布**：发布到 `PUBLIC` / `INTERNAL`（或产品内等价范围）；发布失败时的提示与引导。
- **与校验联动**：展示阻塞/警告类校验结果，并支持用户通过编辑把流程修到**可发布**状态。
- **Preview**：在 Builder 内对「研究员将看到的内容」的预览（与已发布快照或等价只读数据对齐）；与模块 1 的体验一致或可解释的差异需在文档或 issue 中说明。

**不负责**

- file2flow 从文件生成初稿的算法与提示词（属模块 3、4）。

---

## 3. 自动建流程（file2flow）— From Scratch

**范围**

- **从零搭建**「文件 / 粘贴文本 → 可编辑初稿图」的端到端能力：入口（上传/粘贴）、与后端的契约、解析与结构化、首版提示词与 JSON 模式、错误处理与可观测性（日志/错误码）。
- 与内部 **Flow / Node / Edge** 模型对齐的**首版**映射与最小可用路径（MVP）。
- 与模块 2 约定：**初稿进入 Builder 后的 handoff**（字段、校验失败时的展示责任边界）。

**与模块 4 的边界**

- 模块 3 侧重 **0→1**：管线存在、能跑通、可演示。
- 模块 4 侧重 **1→N**：在已有实现上改质量、改边界情况，不负责推翻重写整条管线（除非与模块 3 协商迁移）。

---

## 4. 自动建流程（file2flow）— Modify

**范围**

- 在**已有 file2flow 实现**上持续迭代：**提示词优化**、输出结构修复、与 `validateFlow` / 业务规则对齐、难例与回归用例集。
- **准确度与鲁棒性**：长文本、模糊输入、多轮修复（如 JSON repair）、失败重试与对用户可见的错误说明（与模块 2 弹窗/文案接口对齐）。
- 性能与成本（token、超时）等非功能性优化可归本模块，但若涉及大改上传管道需与模块 3 或模块 2 协调。

**不负责**

- Researcher 端主流程 UI（属模块 1）；Builder 画布交互本体（属模块 2）。

---

## 5. 用户系统（暂缓）

**状态**

- **本期不做**：不排开发、不阻塞模块 1–4；相关需求（登录、多租户、协作邀请）以占位或 demo 账号为主即可。

**未来启用时**

- 账号与登录、数据库归属、协作 vs 分发分享、邀请链接与权限模型等，仍适用下文「SSO 与简化替代方案」中的架构原则（认证可插拔、应用内用户与授权稳定）。

---

## 未来 SSO 与「现阶段做不了 SSO」时的简化 / 替代方案

（供**未来恢复模块 5** 或做技术预研时参考；当前模块 5 暂缓，不要求本期落地。）

长期目标若是 **SSO**（学校/机构 IdP：OIDC / SAML 等），在拿不到 IdP 前仍可先做**应用内用户与权限模型**，并把登录做成可替换层。

### 原则

1. **认证（Authentication）可插拔**：今天简化登录，明天换 SSO，只换这一层。
2. **授权与归属（Authorization + tenancy）保持稳定**：`user`、`workspace`、`flow` 归属、`membership`、分享规则尽量不随 IdP 推倒重来。
3. **外部身份预留映射字段**：如 `external_subject`（OIDC `sub`）；接 SSO 后写入。

### 推荐的简化替代（无 SSO 阶段）

| 能力 | 建议 | 说明 |
|------|------|------|
| 注册 / 登录 | 邮箱 + 密码，或 Magic link，或演示邀请码 | 不依赖学校 IdP |
| 会话 | HttpOnly Cookie + Session，或短期 JWT | 与 OIDC 回调后发会话形态一致 |
| 用户主键 | 内部 `user_id`（UUID）不变 | SSO 后 `upsert` 绑定 |
| 协作分享 | 链接邀请 + 角色 + 可选过期 | 与 IdP 无关 |

### 接 SSO 时的典型迁移

在登录侧增加 OIDC 授权码流程 → 回调绑定 `external_subject` → 逐步弱化本地密码；**不必**从零重做 flow 归属与邀请模型。

## 模块间依赖（简表）

| 依赖方向 | 说明 |
|----------|------|
| 1 ← 2 | 发布成功、快照进入知识库后，Researcher 才有稳定数据源；Preview 应与 1 的体验对齐或明确差异。 |
| 2 ← 3、4 | file2flow（初稿或迭代后的输出）进入 Builder 编辑与发布闭环；校验失败文案与 handoff 由三方约定。 |
| 4 → 3 | 若「From Scratch」管线尚未就绪，Modify 可基于原型代码迭代；管线架构大变更需与模块 3 同步。 |
| 5 | 暂缓；1–4 不依赖本期用户系统交付。 |
